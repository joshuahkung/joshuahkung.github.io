Kicked off the project this week by building out the first working coordinator simulations: a pick-up basketball run and a Knicks watch party, both run through a shared dashboard that tracks the message thread, RSVPs, and the coordinator's evolving policy log.

The basketball run started simple — I (as organizer) message a group chat proposing a time and court, and the system tracks who's confirmed, who's still awaiting reply, and whether we have enough people to actually play. The dashboard surfaces this as a live "who's in" table alongside the raw message thread, so I can see both the human-readable conversation and the underlying state at the same time.

The Knicks watch party sim added a second axis: not just RSVP status, but a few explicit questions baked into the first outreach message — are you coming, where should we watch, and is cost a concern. This forced me to start thinking about the coordinator as something that needs a **custom policy log**, not just a message thread. Right now that log records things like "initial contact" and tracks whether a policy was drafted, needs action, or was approved and sent — the beginnings of an audit trail for what the system decided and why.

Both sims run on the same underlying pattern: the coordinator (an AI agent) drafts messages, and I approve or edit them before they go out. That approval step is deliberate — I want a clear boundary between the system proposing an action and the system taking one autonomously, which is the core idea I'm building toward for the whole project.

**Next steps:** flesh out the policy log so it captures more than "no action" / "drafted" — I want it to start explaining *why* a decision was made, and add real logic for handling things like weather (does rain force an indoor venue?) and court/logistics decisions instead of leaving them fully manual.
