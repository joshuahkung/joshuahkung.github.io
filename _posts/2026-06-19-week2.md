This week was about turning the coordinator from something that just tracks state into something that actually *decides* things — starting with three concrete policies.

**P3 — no-response assumption.** If an invitee hasn't replied, the system assumes they're busy rather than declining. It won't mark someone "no" unless they've explicitly said so, and it won't nag people who've already answered every question asked of them — it only follows up with people who still have outstanding asks.

**P4 — weather check.** The system checks the forecast for the proposed run day. If the weather looks bad (rain, heavy wind), indoor becomes the only acceptable option, and the coordinator drafts a raincheck message rather than plowing ahead outdoors. If the weather's fine, it still prefers outdoor but treats indoor as acceptable too.

**P5 — court decision + travel.** Once enough people have voted on a court (all votes in, or after a couple of follow-up cycles), the system computes rough travel time for each person from their neighborhood to the proposed location and includes a one-line travel summary in the draft message. It also prefers free indoor courts around the city and flags anything that requires payment, so cost surprises get caught before they go out in a message.

Also rebuilt the dashboard UI this week — moved to a cleaner dark-mode layout that separates run info (organizer, day, time, court), a live player list with RSVP status and location, and the full message thread, all rebuilt fresh on every run so the state I'm looking at always reflects the latest messages.

By the end of the week the basketball run sim had a full round-trip working: propose a time → collect RSVPs → check weather → decide on a court factoring in travel → confirm with the group. Stephen, Igor, and Jadon all showed up as live test participants with real back-and-forth in the thread, which made it easier to see where the system's assumptions (like P3) actually get tested against real ambiguous responses.

**Next steps:** the policies so far are all pre-run logistics (weather, travel, court). I want to start handling the harder case — mid-run problems, like someone flaking last-minute or a new person showing up wanting something the system has no policy for yet.
